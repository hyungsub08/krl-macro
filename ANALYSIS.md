# 코레일 자동 예매 매크로 - 기술 분석 및 방어 대책

## 1. 매크로 동작 원리

### 1.1 전체 흐름

```
사용자 입력 (웹 UI)
    ↓
Flask 서버 (app.py)
    ↓
korail2 라이브러리 (+ DynaPath anti-bot 우회 패치)
    ↓
코레일 모바일 API 서버
    ↓
반복: 조회 → 좌석 발견 → 예매 시도
    ↓
예매 성공 시 종료
```

### 1.2 핵심 기술 구성

| 계층 | 기술 | 역할 |
|---|---|---|
| **프론트엔드** | HTML/CSS/JavaScript | 사용자 입력 수집, 실시간 로그 표시 (Polling 기반) |
| **백엔드** | Flask (Python) | REST API 제공, 매크로 세션 관리 |
| **비동기 처리** | threading | 매크로를 백그라운드 스레드에서 실행 |
| **코레일 통신** | korail2 라이브러리 (패치 적용) | 코레일 모바일 앱 API를 역공학하여 HTTP 요청 수행 |
| **Anti-bot 우회** | DynaPathMasterEngine | `x-dynapath-m-token` 헤더 및 `Sid` 파라미터 생성 |
| **상태 관리** | Polling (1초 간격) | 프론트엔드가 `/api/status`를 주기적으로 호출하여 진행 상황 확인 |

### 1.3 korail2 라이브러리의 동작 방식

korail2는 코레일 **모바일 앱이 사용하는 비공개 REST API**를 역공학(reverse engineering)하여 만들어진 오픈소스 Python 라이브러리다.

#### 로그인 과정
1. `/classes/com.korail.mobile.common.code.do`에서 AES 암호화 키를 동적으로 수신
2. 수신된 키로 사용자 비밀번호를 **AES-CBC 암호화** 후 base64 이중 인코딩하여 전송
3. **DynaPath 토큰**(`x-dynapath-m-token`)을 헤더에, **Sid**(AES-CBC 암호화된 디바이스+타임스탬프)를 POST body에 포함
4. 코레일 서버로부터 세션 쿠키 및 `Key` 수신
5. 이후 모든 요청에 세션 쿠키를 포함하여 인증 유지

#### 열차 조회
- 코레일 모바일 API 엔드포인트(`ScheduleView`)에 출발역, 도착역, 날짜, 시간 등의 파라미터를 담아 **HTTP POST** 요청
- 보호 대상 경로이므로 `x-dynapath-m-token` 헤더 및 `Sid` 파라미터 자동 포함
- 응답으로 열차 목록(JSON)을 수신
- 각 열차의 좌석 상태(`general_seat`, `special_seat`) 필드로 매진 여부 판별
  - `'11'` = 예약 가능, `'13'` = 매진, `'00'` = 해당 좌석 없음

#### 예매
- 조회 결과에서 얻은 열차 정보를 예매 API(`TicketReservation`)에 전송
- 역시 `x-dynapath-m-token` + `Sid` 포함 필요
- 성공 시 예약번호, 결제 기한, 금액 등을 응답으로 수신

### 1.4 매크로의 핵심 전략

```python
while True:
    # 매 요청마다 새로운 DynaPath 토큰 + Sid 생성
    trains = search(include_no_seats=True)  # 매진 포함 전체 조회
    for train in trains:
        if train.has_seat():                # 좌석 발생 감지
            reserve(train)                  # 즉시 예매
            return                          # 성공 시 종료
    sleep(interval)                         # 대기 후 재시도
```

- **`include_no_seats=True`**: 매진된 열차도 포함하여 조회함으로써, 취소표 발생 시 즉시 감지
- **짧은 조회 간격**: 사람이 수동으로 새로고침하는 것보다 훨씬 빠르게 반복 조회
- **자동 예매**: 좌석 발견 즉시 예매 API를 호출하여 사람의 반응 시간을 제거
- **자동 재로그인**: 세션 만료 시 자동으로 재인증하여 중단 없이 동작
- **요청별 토큰 갱신**: 매 API 호출마다 새로운 `x-dynapath-m-token`과 `Sid`를 생성하여 anti-bot 검증 통과

## 2. DynaPath Anti-bot 우회 상세 분석

### 2.1 DynaPath 도입 배경

2025년 초, 코레일은 앱 업데이트를 통해 **DynaPath** 보안 솔루션을 도입했다. 이후 비공식 클라이언트(korail2 등)로 API를 호출하면 다음과 같은 응답이 반환되기 시작했다:

```
status_code=200
strResult=FAIL
h_msg_cd=MACRO ERROR
h_msg_txt=원활한 서비스 이용을 위해 앱을 최신 버전으로 업데이트한 뒤 
          재실행 후 안정적인 환경에서 사용해 주시기 바랍니다.
```

### 2.2 DynaPath 보호 대상 경로

서버 측에서 `x-dynapath-m-token` 헤더를 검증하는 6개 API 경로:

| 경로 | 기능 |
|---|---|
| `/classes/com.korail.mobile.login.Login` | 로그인 |
| `/classes/com.korail.mobile.seatMovie.ScheduleView` | 열차 조회 |
| `/classes/com.korail.mobile.seatMovie.ScheduleViewSpecial` | 특실 열차 조회 |
| `/classes/com.korail.mobile.certification.TicketReservation` | 예매 |
| `/classes/com.korail.mobile.nonMember.NonMemTicket` | 비회원 예매 |
| `/classes/com.korail.mobile.trn.prcFare.do` | 운임 조회 |

### 2.3 역공학 과정

#### 1단계: 트래픽 캡처

공격자는 **mitmproxy**, **Charles Proxy**, **Burp Suite** 등의 MITM 프록시를 사용하여 코레일 앱의 HTTP 트래픽을 캡처했다. 안드로이드 기기에 프록시 CA 인증서를 설치하고, 앱의 네트워크 요청을 가로채 분석했다.

이 과정에서 다음을 확인할 수 있었다:
- 보호 경로 요청에 `x-dynapath-m-token` 헤더가 포함됨
- 토큰은 `bEeEP`로 시작하는 긴 문자열
- 매 요청마다 토큰이 변경됨 (동적 생성)
- `Sid`라는 추가 파라미터가 POST body에 포함됨

> **참고**: `nf*.letskorail.com` 도메인에는 인증서 피닝이 적용되어 트래픽 확인이 불가능했으나, 메인 API 서버(`smart.letskorail.com`)에는 피닝이 미적용되어 캡처가 가능했다.

#### 2단계: 앱 디컴파일 및 코드 분석

캡처된 토큰의 구조를 이해하기 위해 코레일 앱(APK)을 디컴파일하여 DynaPath SDK의 토큰 생성 로직을 분석했다:

- **jadx** 또는 **apktool** 등으로 APK를 디컴파일
- DynaPath SDK 관련 클래스(인코딩 테이블, 키 생성 등)를 식별
- 토큰 생성에 사용되는 상수값과 알고리즘을 추출

#### 3단계: 토큰 생성 로직 재구현

추출된 로직을 Python으로 재구현한 것이 `DynaPathMasterEngine` 클래스이다.

### 2.4 DynaPath 토큰 생성 구조 (상세)

#### 토큰의 전체 구조

```
bEeEP + [길이 표시 문자] + [키 인코딩] + [본문 인코딩]
  │          │                  │              │
  │          │                  │              └─ 평문을 커스텀 테이블로 인코딩
  │          │                  └─ 동적 키를 기본 테이블로 인코딩
  │          └─ 기본 테이블에서 키 인코딩 길이에 해당하는 문자
  └─ 매직 프리픽스 (고정)
```

#### 입력값 (평문)

토큰에 인코딩되는 평문은 다음과 같은 디바이스/앱 메타데이터이다:

```
ai=com.korail.talk          ← 앱 패키지명
di=558a4f02041657ea         ← 디바이스 ID (하드코딩)
as=%5B38ff229cb34c...%5D    ← 앱 서명 해시 (하드코딩)
su=false                    ← 슈퍼유저(루팅) 여부
dbg=false                   ← 디버그 모드 여부
emu=false                   ← 에뮬레이터 여부
hk=false                    ← 후킹 탐지 여부
it=1711234567890            ← 앱 시작 타임스탬프 (ms)
ts=1711234568123            ← 현재 타임스탬프 (ms)
rt=0                        ← 리트라이 횟수
os=13                       ← Android OS 버전
dm=SM-S928N                 ← 디바이스 모델
st=Android                  ← OS 타입
sv=v1                       ← SDK 버전
```

#### 인코딩 알고리즘

**기본 치환 테이블** (62문자):
```
3FE9jgRD4KdCyuawklqGJYmvfMn15P7US8XbxeLQtWT6OicBAopINs2Vh0HZrz
```

**인코딩 과정**:

```
① 문자열을 커스텀 바이트 배열로 변환 (string2xA1s)
   - ASCII(< 128): 그대로 1바이트
   - 2바이트 유니코드: 2바이트로 분할
   - 3바이트 유니코드: 3바이트로 분할
   
② 바이트 쌍(2개씩)을 161진수로 합산
   val = byte1 * 161 + byte2
   
③ 합산값을 30진수로 분해하여 3자리 추출
   digit[0] = val % 30
   digit[1] = (val / 30) % 30
   digit[2] = (val / 900) % 30
   
④ 각 자릿수를 치환 테이블의 인덱스로 사용하여 문자로 변환

결과: 입력 2바이트 → 출력 3문자 (약 1.5배 확장)
```

#### 동적 키 메커니즘

매 요청마다 새로운 토큰을 생성하기 위해 **동적 키**를 사용한다:

```python
# 1. 동적 키 생성 (매 요청마다 변경)
rand = "AB3X"                    # 랜덤 4자리 (대문자+숫자)
ts = 1711234568123               # 현재 타임스탬프 (ms)
dyn_key = f"v1+{rand}+{ts}"     # 예: "v1+AB3X+1711234568123"

# 2. 키를 기본 테이블로 인코딩
key_enc = encode(dyn_key, BASE_TABLE)

# 3. 키에서 정수값 파생 (make_key)
#    - 각 문자의 유니코드 코드포인트에서 최상위 비트 위치를 계산
#    - 이를 누적 곱셈으로 결합하여 큰 정수 생성
big_key = make_key(dyn_key)

# 4. 정수값으로 커스텀 인코딩 테이블 생성 (make_encode_table)
#    - 기본 62문자 테이블에서 big_key를 기반으로 30문자를 선택
#    - 매 요청마다 다른 순서의 테이블이 생성됨
custom_table = make_encode_table(big_key, 30, BASE_TABLE)

# 5. 평문을 커스텀 테이블로 인코딩
body_enc = encode(plaintext, custom_table)

# 6. 최종 토큰 조립
token = f"bEeEP{BASE_TABLE[len(key_enc)]}{key_enc}{body_enc}"
```

이 방식의 핵심은 **매 요청마다 랜덤값과 타임스탬프가 바뀌므로 키가 달라지고, 키가 달라지면 인코딩 테이블이 달라져서 동일한 평문이라도 매번 다른 토큰이 생성된다**는 것이다.

#### Sid 생성

`Sid`는 서버가 요청의 디바이스와 시간을 검증하기 위한 추가 파라미터이다:

```python
# 평문: 디바이스 코드 + 타임스탬프 (밀리초)
plaintext = f"AD{timestamp_ms}"     # 예: "AD1711234568123"

# AES-CBC 암호화 (키와 IV가 동일)
key = iv = b"2485dd54d9deaa36"      # 16바이트 하드코딩 키
cipher = AES.new(key, AES.MODE_CBC, iv=key)
encrypted = cipher.encrypt(pad(plaintext.encode(), 16))

# Base64 인코딩 + 개행문자
Sid = base64.b64encode(encrypted).decode() + "\n"
```

### 2.5 우회가 성공한 근본 원인

| 취약점 | 설명 |
|---|---|
| **클라이언트 측 토큰 생성** | 토큰 생성 로직이 전적으로 앱 내부에 존재하며, 서버는 생성된 토큰의 형식만 검증한다. 서버 참여 없이 클라이언트만으로 토큰을 생성할 수 있는 구조이므로, 로직을 알면 누구나 유효한 토큰을 만들 수 있다. |
| **하드코딩된 비밀값** | 인코딩 테이블(`3FE9jgRD4K...`), AES 키(`2485dd54d9deaa36`), 앱 서명(`AS_VALUE`), 디바이스 ID 등이 모두 앱 바이너리에 하드코딩되어 있다. APK 디컴파일로 즉시 추출 가능하다. |
| **앱 무결성 미검증** | 토큰에 `su=false`, `emu=false`, `hk=false` 등 환경 정보가 포함되지만, 이 값들은 클라이언트가 자체 보고하는 것이므로 얼마든지 위조 가능하다. **Google Play Integrity API** 등 외부 검증이 없어 실제 앱 환경인지 확인할 방법이 없다. |
| **메인 API 서버 MITM 가능** | `nf*.letskorail.com`에만 인증서 피닝이 적용되었고, 실제 API 서버(`smart.letskorail.com`)에는 미적용되어 트래픽 캡처가 가능했다. |
| **고정된 알고리즘** | 인코딩 알고리즘(base-30 + 동적 테이블)이 앱 업데이트 없이는 변경되지 않으므로, 한 번 역공학하면 장기간 유효하다. |
| **서버 측 심층 검증 부재** | 서버가 토큰 내 타임스탬프의 유효 범위, `device_id`의 실존 여부, `app_start_ts`와 세션의 일관성 등을 충분히 검증하지 않는 것으로 보인다. |

## 4. 방어 대책

### 4.1 API Rate Limiting (요청 속도 제한)

**문제**: 매크로는 1초 간격으로 반복 조회가 가능하다.

**대책**:
- 동일 사용자(세션/IP/계정) 기준으로 **분당 요청 횟수 제한**
  - 예: 조회 API는 분당 10회, 예매 API는 분당 3회
- 제한 초과 시 429 (Too Many Requests) 응답 반환
- **점진적 차단**: 반복 위반 시 차단 시간을 지수적으로 증가 (1분 → 5분 → 30분)

```
구현 예시:
- Token Bucket 또는 Sliding Window 알고리즘 적용
- Redis 기반 분산 Rate Limiter로 서버 간 공유
```

### 4.2 CAPTCHA 도입

**문제**: 매크로는 사람의 개입 없이 자동으로 동작한다.

**대책**:
- 열차 조회 N회 이상 시 **CAPTCHA 인증** 요구
- 예매 직전에 CAPTCHA 필수 적용
- reCAPTCHA v3 (점수 기반)를 활용하여 봇 의심 트래픽을 자동 차단
- 반복 실패 시 이미지 기반 CAPTCHA로 전환

### 4.3 DynaPath 토큰 검증 강화 (현재 우회된 방어 계층)

**현황**: 코레일이 2025년 초 도입한 DynaPath SDK는 역공학으로 우회되었다. 근본 원인은 토큰 생성이 전적으로 클라이언트에서 이루어지고, 서버는 형식 검증만 수행하기 때문이다.

**강화 대책**:
- **서버 측 토큰 검증 강화**:
  - 토큰 내 타임스탬프의 유효 윈도우를 좁게 설정 (예: ±30초)
  - 동일 `device_id`에서의 요청 빈도 모니터링
  - 토큰 내 `app_start_ts`와 실제 세션 시작 시간의 일관성 검증
- **토큰 생성 키 동적 교체**:
  - 인코딩 테이블과 AES 키를 앱 업데이트 시마다 변경
  - 서버에서 주기적으로 새 키를 배포하고, 이전 키를 일정 시간 후 폐기
  - 이렇게 하면 역공학된 코드가 앱 업데이트마다 무력화됨
- **Challenge-Response 방식 도입**:
  - 서버가 매 요청마다 고유한 challenge(nonce)를 발급
  - 클라이언트가 challenge를 포함하여 토큰을 생성
  - 서버가 자신이 발급한 challenge와 대조하여 검증
  - 이렇게 하면 클라이언트가 독립적으로 토큰을 생성할 수 없음

### 4.4 앱 무결성 검증 (App Attestation)

**문제**: 현재 DynaPath 토큰에 포함된 환경 정보(`su`, `emu`, `hk`)는 클라이언트가 자체 보고하므로 위조 가능하다.

**대책**:
- **Android: Play Integrity API**
  - Google 서버가 앱 바이너리의 무결성과 실행 환경을 직접 검증
  - 역공학된 Python 클라이언트는 유효한 attestation 토큰을 생성할 수 없음
- **iOS: App Attest / DeviceCheck**
  - Apple 서버를 통한 앱 무결성 증명
- 이를 통해 비공식 클라이언트를 **원천적으로** 차단 가능

### 4.5 TLS Certificate Pinning 강화

**문제**: mitmproxy 등으로 앱 트래픽을 분석하여 API 구조와 토큰을 캡처할 수 있다. 현재 `nf*.letskorail.com`에만 피닝이 적용되어 있고, 메인 API 서버에는 미적용이다.

**대책**:
- **모든 API 엔드포인트**에 TLS Certificate Pinning 적용
- **Network Security Config**(Android) / **ATS**(iOS) 강화
- 루팅/탈옥 탐지와 연계하여, 비정상 환경에서의 피닝 우회 시도 차단
- 피닝 우회 탐지 시 세션 즉시 무효화

### 4.6 TLS 핑거프린팅 (JA3/JA4)

**문제**: 매크로는 Python `requests` 라이브러리로 API를 호출한다. User-Agent는 `Android 13 / SM-S928N`으로 위장하지만, TLS 핸드셰이크 패턴은 Python/urllib3의 것이다.

**대책**:
- 서버 측에서 **JA3/JA4 핑거프린트**를 수집하여 User-Agent와 대조
- User-Agent가 Android인데 TLS 핑거프린트가 Python인 경우 → 봇으로 판정
- 이 방법은 **클라이언트 코드 변경 없이 서버 측만으로** 탐지 가능

### 4.7 행동 패턴 분석 (Anomaly Detection)

**문제**: Rate Limiting만으로는 느린 간격의 매크로를 탐지하기 어렵다.

**대책**:
- **정상 사용자 패턴 모델링**:
  - 조회 간격의 분산 (사람은 불규칙, 매크로는 일정)
  - 조회 후 예매까지의 반응 시간 (매크로는 수 ms, 사람은 수 초)
  - 세션 지속 시간 (매크로는 장시간 연속 조회)
  - `app_start_ts`와 요청 패턴의 일관성 (매크로는 앱 시작 후 즉시 반복 조회)
- **이상 탐지 규칙 예시**:
  - 조회 간격의 표준편차가 0.1초 미만 → 봇 의심
  - 조회 후 예매까지 500ms 미만 → 봇 의심
  - 동일 구간 연속 조회 100회 이상 → 경고
  - 동일 `device_id`에서 비정상적 빈도의 토큰 생성 → 봇 의심
- 의심 세션에 대해 CAPTCHA 발동 또는 일시 차단

### 4.8 예매 정책 강화

**문제**: 대량 선점 및 미결제로 다른 사용자 피해 발생.

**대책**:
- **동시 미결제 예약 제한**: 1인당 미결제 예약 최대 1~2건
- **결제 시간 단축**: 예약 후 결제 기한을 20분 → 10분으로 축소
- **반복 미결제 패널티**: 일정 기간 내 미결제 취소 N회 초과 시 예매 제한
- **동일 구간 중복 예매 제한**: 같은 날짜/구간에 대해 중복 예약 불가

### 4.9 대기열 시스템 도입

**문제**: 선착순 방식은 속도 경쟁에서 매크로가 유리하다.

**대책**:
- **가상 대기열**: 인기 구간/시간대에 대기열 도입
  - 조회 요청 시 대기열에 진입 → 순번 도달 시 예매 가능
  - 대기 중 주기적으로 CAPTCHA 또는 인터랙션 요구
- **랜덤 배정**: 취소표 발생 시 즉시 공개하지 않고, 일정 시간(예: 30초) 동안 모인 요청 중 랜덤 배정
  - 속도 우위를 무력화

## 6. 요약

이 매크로는 **비공개 모바일 API의 역공학**, **DynaPath anti-bot 토큰 생성 로직 재구현**, **반복 자동 조회**, **즉시 예매**라는 네 가지 요소를 결합하여 동작한다.

코레일이 2025년 초 도입한 DynaPath SDK 기반 anti-bot 방어는 다음의 근본적 한계로 우회되었다:
- 토큰 생성 로직이 **클라이언트에만** 존재하고 서버는 형식만 검증
- 인코딩 테이블, AES 키 등 **비밀값이 앱에 하드코딩**
- **앱 무결성 검증(App Attestation)** 미적용
- 메인 API 서버에 **TLS 피닝 미적용**으로 트래픽 분석 가능

이는 **"클라이언트에 있는 비밀은 비밀이 아니다"** 라는 보안 원칙을 잘 보여준다. 가장 효과적인 방어는 단일 대책이 아닌 **다층 방어(Defense in Depth)** 전략이며, TLS 핑거프린팅과 토큰 키 교체로 즉각적인 효과를 얻고, App Attestation과 행동 분석으로 장기적인 방어 체계를 구축하는 것을 권장한다.

## 7. FAQ

### Q1. 이 매크로는 korail2 라이브러리가 있어야만 동작하나요?

**아닙니다.** korail2는 코레일 모바일 앱의 비공개 REST API를 Python으로 감싼 편의 래퍼(wrapper)입니다. 이 라이브러리가 하는 일은:

1. 코레일 앱이 서버에 보내는 HTTP 요청 형식을 그대로 재현
2. 비밀번호 AES 암호화 처리
3. 응답 JSON 파싱

이 모든 것은 Python `requests` 라이브러리만으로도, 심지어 `curl`이나 다른 언어(Java, Node.js, Go 등)로도 동일하게 구현할 수 있습니다. korail2는 개발 시간을 단축하기 위한 도구일 뿐, 필수 요소가 아닙니다.

### Q2. korail2 없이 직접 구현하려면 무엇이 필요한가요?

korail2가 감싸고 있는 코레일 서버의 실제 API 엔드포인트는 다음과 같습니다:

| 기능 | 메서드 | URL |
|---|---|---|
| 로그인 | POST | `https://smart.letskorail.com:443/classes/com.korail.mobile.login.Login` |
| 열차 조회 | POST | `https://smart.letskorail.com:443/classes/com.korail.mobile.seatMovie.ScheduleView` |
| 예매 | GET | `https://smart.letskorail.com:443/classes/com.korail.mobile.certification.TicketReservation` |
| 취소 | GET | `https://smart.letskorail.com:443/classes/com.korail.mobile.reservationCancel.ReservationCancelChk` |

직접 구현 시 추가로 필요한 것:

- 비밀번호 AES-CBC 암호화 (암호화 키는 서버에서 동적 발급)
- DynaPath anti-bot 토큰 생성 (`x-dynapath-m-token` 헤더)
- Sid 파라미터 생성 (AES-CBC 암호화)
- 요청 파라미터 형식 (korail2 소스 코드에서 확인 가능)

### Q3. korail2를 차단하면 매크로를 막을 수 있나요?

**막을 수 없습니다.** korail2는 특별한 기술이 아니라 공개된 HTTP 프로토콜 위에서 동작합니다. 라이브러리를 차단하더라도 동일한 요청을 직접 구성하여 보낼 수 있으며, 서버 입장에서 korail2의 요청과 공식 앱의 요청을 구분할 방법이 제한적입니다.

따라서 매크로를 근본적으로 차단하려면 **서버 측 방어**(TLS 핑거프린팅, App Attestation, 행동 패턴 분석, 대기열 시스템 등)가 필요합니다. 자세한 내용은 본 문서의 [4. 방어 대책](#4-방어-대책) 섹션을 참조하세요.

### Q4. 실제로 예매가 되나요?

**됩니다.** 테스트에서 실제 코레일 서버를 통해 예약이 생성되었으며, 예약번호와 금액이 정상적으로 발급되었습니다.

| 테스트 | 예약번호 | 열차 | 금액 | 결과 |
|---|---|---|---|---|
| 1차 | `320260419021093` | KTX 015호 서울→부산 | 59,800원 | 예매 성공, 기한만료 자동취소 |
| 2차 | `320260419241827` | KTX 011호 서울→부산 | 83,700원 | 예매 성공, 기한만료 자동취소 |
| 3차 | `320260419294600` | KTX 011호 서울→부산 | 56,800원 | 예매 성공, 즉시 취소 성공 |

예매 후 결제를 하지 않으면 약 20분 후 자동 취소됩니다. 매크로는 예매까지만 수행하며, 결제는 사용자가 코레일 앱/홈페이지에서 직접 해야 합니다.

### Q5. DynaPath anti-bot이 적용되어 있는데 어떻게 우회가 가능한가요?

DynaPath SDK의 토큰 생성 로직이 **클라이언트(앱) 내부에만** 존재하고, 서버는 생성된 토큰의 형식만 검증하기 때문입니다. 앱을 디컴파일하여 토큰 생성 알고리즘과 하드코딩된 키를 추출한 뒤, 이를 Python으로 재구현하면 서버가 구분할 수 없는 유효한 토큰을 생성할 수 있습니다.

자세한 역공학 과정과 토큰 구조는 본 문서의 [2. DynaPath Anti-bot 우회 상세 분석](#2-dynapath-anti-bot-우회-상세-분석) 섹션을 참조하세요.

### Q6. 코레일이 알고리즘을 변경하면 매크로가 무력화되나요?

**일시적으로는 그렇습니다.** 코레일이 다음을 변경하면 현재 매크로는 동작을 멈춥니다:

- DynaPath 인코딩 테이블 또는 알고리즘 변경
- AES 키 변경
- API 엔드포인트 또는 파라미터 형식 변경
- 앱 버전 검증 강화

그러나 앱 업데이트가 배포되면 다시 디컴파일하여 변경된 로직을 파악할 수 있으므로, **장기적으로는 App Attestation(Play Integrity API) 등 근본적인 대책이 필요**합니다.

### Q7. 매크로의 조회 간격은 어느 정도가 안전한가요?

> **업데이트 (2026-04-14)**: 1초 간격으로 예매를 시도할 경우 코레일 서버에서 차단되는 것이 확인되었습니다. **최소 3초 이상의 간격을 권장합니다.**

테스트 결과:

| 간격 | 연속 횟수 | 결과 |
|---|---|---|
| **1초** | — | **예매 시도 시 차단됨** (Rate Limiting 탐지) |
| 3초 | 20회 | 전부 성공, Rate Limiting 미탐지 |
| 30초 | 9회 (4분간) | 전부 성공, 세션 만료 없음 |

**권장 설정: 3~5초**

- 조회(ScheduleView)와 예매(TicketReservation) 각 요청 사이에 3초 이상 간격을 두어야 합니다.
- 1초 간격은 Rate Limiting에 걸려 예매 요청이 차단되며, 연속 차단 시 세션이 만료될 수 있습니다.
- 현재 3초 간격에서는 서버의 Rate Limiting이 탐지되지 않으나, 이는 언제든 변경될 수 있습니다.
- 웹 UI 기본값은 **3.0초**로 설정되어 있습니다.
